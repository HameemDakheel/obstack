# v1.1 Standard Profile Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship obstack v1.1 — Standard profile overlay, OTel host metrics + Host Health dashboard, four opt-in alert packs (Postgres/Nginx/Redis/Host), and a redesigned external-client demo under `examples/otel-demo/`.

**Architecture:** Strictly additive on top of v1.0.1 (the rename release). Compose overlay pattern continues (`compose/standard.yml` joins `compose/simple.yml`). Host metrics extend the existing OTel Collector — no new container. Alert packs are drop-in YAMLs the user copies into `alerts/`. Demo is rebuilt as a standalone compose project that talks to obstack via the public OTLP endpoint with HTTP Basic auth.

**Tech Stack:** Docker Compose, OTel Collector contrib (`hostmetrics` receiver), Prometheus rule files, Grafana dashboard JSON (schema 39), Bash, Make. No new images vs v1.0.1.

**Spec reference:** [docs/superpowers/specs/2026-04-25-v1.1-standard-profile.md](../specs/2026-04-25-v1.1-standard-profile.md)
**Predecessor:** `v1.0.1` (the rename release) must be tagged before starting.

```markdown
# Optional alert packs

Each YAML file in this directory is a self-contained Prometheus alert rule pack. They are **not loaded by default**.

## Activate a pack

```bash
cp alerts/optional/postgres.yaml alerts/
docker compose -f docker-compose.yml -f compose/simple.yml restart prometheus
```

## Deactivate a pack

```bash
rm alerts/postgres.yaml
docker compose -f docker-compose.yml -f compose/simple.yml restart prometheus
```

## Available packs

| Pack | Requires | Purpose |
|------|----------|---------|
| `postgres.yaml` | [`prometheus-postgres-exporter`](https://github.com/prometheus-community/postgres_exporter) sending metrics to obstack | Connection saturation, replication lag, deadlocks, long-running transactions |
| `nginx.yaml` | [`nginx-prometheus-exporter`](https://github.com/nginxinc/nginx-prometheus-exporter) | Up/down, connection waiting ratio |
| `redis.yaml` | [`redis_exporter`](https://github.com/oliver006/redis_exporter) | Up/down, memory pressure, evictions, rejected connections |
| `host.yaml` | The OTel Collector `hostmetrics` receiver (built into obstack v1.1+; nothing extra to deploy) | Host CPU saturation, memory pressure, disk fill, iowait, network errors |

## Adding your own packs

Drop a new YAML file (any filename) into `alerts/optional/` for community sharing or `alerts/` for active loading. Validate with `promtool check rules <file>`.
```

- [x] **Step 3: Write `alerts/optional/postgres.yaml`**

```yaml
# obstack optional alert pack: PostgreSQL
# Requires: prometheus-postgres-exporter scraping your Postgres database
# Activate: cp alerts/optional/postgres.yaml alerts/ && restart prometheus

groups:
  - name: obstack-postgres
    interval: 30s
    rules:
      - alert: PostgresDown
        expr: pg_up == 0
        for: 2m
        labels:
          severity: critical
          pack: postgres
        annotations:
          summary: "Postgres instance {{ $labels.instance }} is down"
          description: "pg_up == 0 for the past 2 minutes. The exporter cannot connect to Postgres."

      - alert: PostgresHighConnectionUsage
        expr: sum by (instance) (pg_stat_activity_count) / on(instance) pg_settings_max_connections > 0.8
        for: 5m
        labels:
          severity: warning
          pack: postgres
        annotations:
          summary: "Postgres {{ $labels.instance }} >80% of max_connections in use"
          description: "{{ $value | humanizePercentage }} of max_connections actively used. Risk of connection-refused at the application layer."

      - alert: PostgresReplicationLagHigh
        expr: pg_replication_lag_seconds > 30
        for: 5m
        labels:
          severity: warning
          pack: postgres
        annotations:
          summary: "Postgres replica {{ $labels.instance }} is {{ $value }}s behind primary"
          description: "Replication lag exceeded 30 seconds for 5 minutes. Read replicas may serve stale data."

      - alert: PostgresLongRunningTransaction
        expr: max by (instance, datname) (pg_stat_activity_max_tx_duration{datname!~"template.*"}) > 300
        for: 5m
        labels:
          severity: warning
          pack: postgres
        annotations:
          summary: "Postgres long-running transaction on {{ $labels.instance }}/{{ $labels.datname }}"
          description: "A transaction has been open for {{ $value }} seconds. Holds locks; may bloat tables and prevent VACUUM."

      - alert: PostgresDeadlocksRising
        expr: rate(pg_stat_database_deadlocks[5m]) > 0.5
        for: 5m
        labels:
          severity: warning
          pack: postgres
        annotations:
          summary: "Postgres deadlock rate climbing on {{ $labels.instance }}"
          description: "{{ $value }} deadlocks/sec average over 5 minutes. Investigate transaction ordering."
```

```yaml
# obstack optional alert pack: Nginx
# Requires: nginx-prometheus-exporter scraping your Nginx instance
# Activate: cp alerts/optional/nginx.yaml alerts/ && restart prometheus
#
# NOTE: Open-source nginx + nginx-prometheus-exporter only exposes coarse
# connection metrics (no per-status-code rates). For 5xx-rate alerts you
# need NGINX Plus or a different exporter (e.g., GoAccess, lua-resty-prometheus).

groups:
  - name: obstack-nginx
    interval: 30s
    rules:
      - alert: NginxDown
        expr: nginx_up == 0
        for: 2m
        labels:
          severity: critical
          pack: nginx
        annotations:
          summary: "Nginx instance {{ $labels.instance }} is down"
          description: "nginx_up == 0 for the past 2 minutes."

      - alert: NginxConnectionWaitingHigh
        expr: nginx_connections_waiting / nginx_connections_active > 0.8
        for: 10m
        labels:
          severity: warning
          pack: nginx
        annotations:
          summary: "Nginx {{ $labels.instance }} has high keep-alive waiting ratio"
          description: "{{ $value | humanizePercentage }} of active connections are idle/waiting. May indicate slow clients or upstream stalls."

      - alert: NginxRequestRateDropped
        expr: rate(nginx_http_requests_total[5m]) == 0 and rate(nginx_http_requests_total[1h] offset 1h) > 1
        for: 10m
        labels:
          severity: warning
          pack: nginx
        annotations:
          summary: "Nginx {{ $labels.instance }} request rate dropped to 0"
          description: "Nginx accepted >1 req/s in the previous hour but 0 in the last 5 minutes. Possible upstream outage or DNS failure."
```

```bash
#!/usr/bin/env bash
# demo-up.sh — bring up examples/otel-demo with computed Basic-auth header.
# Reads examples/otel-demo/.env (or .env.example as fallback), base64-encodes
# the credentials, exports DEMO_OBSTACK_AUTH_HEADER, then docker compose up -d.

set -euo pipefail

REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
DEMO_DIR="$REPO_ROOT/examples/otel-demo"

cd "$DEMO_DIR"

if [[ ! -f .env ]]; then
  if [[ -f .env.example ]]; then
    echo "→ Creating .env from .env.example (edit it before re-running for production)"
    cp .env.example .env
  else
    echo "ERROR: no .env or .env.example in $DEMO_DIR" >&2
    exit 1
  fi
fi

# Source vars from .env without leaking quoting.
set -a
# shellcheck disable=SC1091
source .env
set +a

: "${DEMO_BASIC_AUTH_USER:?must be set in .env}"
: "${DEMO_BASIC_AUTH_PASSWORD:?must be set in .env}"
: "${DEMO_OBSTACK_ENDPOINT:?must be set in .env}"

# Compute and export the OTLP header.
B64="$(printf '%s:%s' "$DEMO_BASIC_AUTH_USER" "$DEMO_BASIC_AUTH_PASSWORD" | base64 -w0 2>/dev/null || \
       printf '%s:%s' "$DEMO_BASIC_AUTH_USER" "$DEMO_BASIC_AUTH_PASSWORD" | base64)"
export DEMO_OBSTACK_AUTH_HEADER="authorization=Basic $B64"

echo "→ Starting demo (endpoint=$DEMO_OBSTACK_ENDPOINT, user=$DEMO_BASIC_AUTH_USER)"
docker compose up -d

echo ""
echo "Demo starting (~2 min). Check obstack's Traces Browser dashboard for the service graph."
echo "Frontend: http://localhost:${DEMO_FRONTEND_PORT:-8082}/"
```

## Notes & Gotchas for the engineer

- **Hostmetrics requires `/hostfs` bind mount.** On some restricted hosts (locked-down cloud, certain PaaS) this isn't allowed — the receiver fails to start with permission errors. If a Simple-profile install starts hitting this, document the env-toggle to disable hostmetrics. Out of scope for v1.1 implementation but flag it if you see it during testing.
- **`make demo-up` needs `.env` populated.** The script auto-creates `.env` from `.env.example` if missing, but the user still needs to set `DEMO_BASIC_AUTH_PASSWORD` to match what their obstack expects. Test failures here are usually "you forgot to update `.env`."
- **Resource limit numbers in `compose/standard.yml` are starting points.** Real production Standard users will tune them based on their workload. Document this in `docs/profiles.md` (already done in Task 9).
- **Demo + Standard on the same 8 GB host will OOM under load.** This is documented; the engineer should not try to "make it fit" by trimming the demo further.
- **`prometheus_postgres_exporter` and `redis_exporter` are *not* shipped by obstack.** The alert packs work only if the user is running those exporters and exporting their metrics into obstack via OTLP or remote_write. Documented in pack-file headers.
- **Optional alert packs use both Prometheus-native names** (`pg_*`, `nginx_*`, `redis_*` — from prometheus exporters) **and OTel semantic-convention names** (`system_cpu_utilization` — from hostmetrics receiver). This is correct: hostmetrics emits OTel naming, app exporters emit Prometheus-traditional naming.
- **Old `make demo`** is hard-removed (no deprecation alias) per the user's guidance during brainstorming. CHANGELOG migration guide is the soft landing.
